### 컨셉:

- 흩어져 있는 상품의 속성(신선도, 실제 유통기한, 성분 안전성)과 고객의 생활패턴(재구매 주기, 선호 브랜드)을 결합하여, 적은 질문으로 최적의 구매를 제안하는 AI Agent
- 단순히 인기 상품을 추천하는 것이 아닌, 고객이 직접 비교하기 어려운 정보와 구매 판단 기준으로 재구성해 추천 (개인한테 맞추어진 시뮬레이터)

### 주요기능:

- 상품 정보 및 리뷰 구조화
    - 리뷰와 상세페이지를 AI가 분석하여 구매 결정에 필요한 핵심 속성 추출. 식품의 경우 신선도 후기, 배송상태, 유통기한 관련 언급을 분석, 생활필수품의 경우 성분, 자극 여부, 품질 불만, 자녀 사용 관련 후기를 구조화
        - ex) “수령일 기준 유통기한이 짧다는 후기가 많음”
        - ex) “아이 피부에 자극이 적었다는 후기가 많음”
- 단품 및 대용량 실속 비교
    - 단순 판매가가 아닌 용량, 단위당 가격, 배송비, 예상 사용 기간, 보관 부담을 함께 고려해 단품과 대용량 중 어떤 선택이 더 합리적인지 판단
        - ex) 4인 가족 기준 한 달 사용량을 바탕으로 “대용량이 단위당 가격은 낮지만 예상 사용 기간이 길어 보관 공간이 필요함” 또는 “현재 사용량 기준으로는 단품 2개 구매가 더 적합함”
- 자녀 사용 안전성 체크
    - 자녀가 먹거나 피부에 닿는 상품에 대해 성분, 원재료, 자극 여부, 후기 기반 안전성 키워드를 요약. 이를 통해 고객이 가격만 보고 구매하지 않고, 가족과 자녀에게 안전한 선택인지 빠르게 판단할 수 있는 기준을 제시
- 자연어 기반 의사결정
    - 사용자가 “아이도 먹을 수 있는 신선한 사과랑, 자극 적은 물티슈를 실속 있게 추천해줘”처럼 자연어로 요청하면, AI Agent가 상품 속성과 고객 조건을 바탕으로 적합한 상품 후보를 제안한다. 추천 시에는 단순히 상품 리스트만 보여주는 것이 아니라, 왜 해당 상품이 적합한지와 주의할 점을 함께 설명

### 성공 지표

- 의사결정 시간 단축 - 사용자가 상품 구매 전 후기, 가격, 용량, 유통기한 정보를 비교하는 데 쓰는 시간을 기존 대비 줄일 수 있는지 측정
- 추천 근거 만족도 - 사용자가 AI Agent의 추천 이유에 대한 만족도를 측정
- 구매 전환율 및 재구매율 - AI 요약 또는 추천을 본 사용자가 실제 구매로 이어지는 비율과, 해당 상품을 반복 구매하는 비율 측정
- 구매 후 불만 감소 - 유통기한, 신선도, 품질, 자극 여부 등과 관련된 구매 후 불만 리뷰나 반품 사유가 감소하는지 확인

## 구현 설명
### 데이터
1. `crawler.js`가 `sample-product.hmtl`을 파싱하여 `products.raw.json`에 저장
2. `enrich.js`가 raw데이터를 읽고 OpenAI API로 상품별 요약 생성 후 `product.enriched.json`에 저장
3. contentHash로 변경된 상품만 API로 재호출하고, 나머지는 캐시 재사용

### 백엔드
- Express API가 `products.enriched.json`을 읽어 반환 — 페이지 로딩 시 AI API 호출 없음
- `/product-insights`는 `enriched JSON`에서 aiSummary 필드만 추출해 반환
- `/chat`은 사용자 조건(가족 수, 자녀 여부, 보관 공간, 월 사용량)과 전체 상품 데이터를 프롬프트에 포함해 GPT-4o-mini 호출

### 프론트엔드
- 해시 기반 라우팅으로 SPA 내 상품 상세 페이지 이동
- 챗봇 대화 기록 localStorage 유지
- 추천 상품 배지에 추천 시점 조건 태그 표시, 클릭 시 상세 페이지 이동

## 강조하고 싶은 부분/설계 의도
초기에는 상품 페이지를 불러올 때마다 OpenAI API를 호출하여 각 제품별 AI 요약을 생성하는 방식으로 구현하려고 했습니다. 그러나 이 방식은 페이지 이동이나 새로고침이 발생할 때마다 API 응답을 기다려야 하므로 제품 카드 렌더링이 지연되는 문제가 있었습니다. 이를 완화하기 위해 제품 정보는 먼저 화면에 표시하고, AI 구매 판단 요약 영역은 스켈레톤 UI로 처리하여 사용자가 로딩 상태를 인지할 수 있도록 개선했습니다.

하지만 페이지 전환이나 새로고침마다 계속 AI API를 호출하는 구조는 응답 속도뿐만 아니라 API 비용 증가, rate limit 발생 가능성, 사용자 경험 저하로 이어질 수 있다고 판단했습니다. 따라서 상품을 크롤링하는 시점에 OpenAI API를 활용해 제품별 요약을 미리 생성하고, 그 결과를 저장해두는 방식으로 구조를 변경했습니다. 이후 상품 페이지에서는 저장된 AI 요약 데이터를 불러와 표시하도록 하여 불필요한 API 호출 수를 줄이고, 페이지 로딩 속도를 개선했습니다.

이 방식은 AI 챗봇의 제품 추천 과정에서도 효과적으로 활용될 수 있습니다. 챗봇이 사용자 질문에 맞는 상품을 추천할 때마다 크롤링 원문 전체나 긴 리뷰 데이터를 모두 OpenAI API에 전달하면 토큰 사용량이 증가하고 응답 속도도 느려질 수 있습니다. 따라서 미리 생성해 둔 제품별 AI 요약을 활용하고, 사용자 질문과 관련 있는 후보 상품만 선별해 API에 전달하도록 구성했습니다. 이를 통해 챗봇은 상품명, 가격, 카테고리, 핵심 요약 정보를 바탕으로 더 빠르게 후보 상품을 비교할 수 있으며, API 비용과 토큰 사용량도 줄일 수 있습니다.

또한 챗봇 기능의 경우, 기존에는 새로고침이나 페이지 이동 시 이전 대화 내용이 사라지는 문제가 있었습니다. 이를 해결하기 위해 대화 기록을 브라우저의 localStorage에 저장하도록 구현했습니다. 그 결과 사용자가 페이지를 이동하거나 새로고침하더라도 이전 대화 흐름이 유지되어, 더 자연스럽게 쇼핑 상담을 이어갈 수 있도록 개선했습니다.

## 아쉬웠던 점 / 개선 여지
솔직히 이번 프로젝트에서 Node.js를 사용한 것은 처음이었습니다. 이전 개인 프로젝트에서는 Spring Boot를 공부해 사용했고, 학교 프로젝트에서는 Django를 주로 다뤘기 때문에 처음에는 익숙하지 않은 기술 스택에 다소 당황했습니다. 하지만 예제들을 찾아보며 학습한 결과, 요청을 받고 데이터를 처리한 뒤 응답을 반환하는 전체적인 백엔드 로직의 흐름은 기존에 경험했던 프레임워크들과 크게 다르지 않다는 것을 알게 되었습니다. 또한 JavaScript를 어느 정도 사용해본 경험이 있었기 때문에, 늦지 않게 프로젝트에 필요한 기능을 구현해 나갈 수 있었습니다.

크롤링 역시 이번 프로젝트에서 처음 다뤄본 영역이었습니다. 실제 쇼핑몰 크롤링 대신 `sample-products.html`을 가상의 크롤링 대상으로 설정하고, Cheerio를 활용해 상품명, 가격, 설명, 리뷰 등의 정보를 파싱한 뒤 JSON 형태로 구조화하는 방식으로 구현했습니다. 이후 해당 데이터를 AI 요약 및 챗봇 추천 기능에서 활용할 수 있도록 정리하는 데 초점을 맞췄습니다.

다만 시간이 더 여유로웠다면 Node.js의 구조와 Express의 동작 방식, 미들웨어 처리, 비동기 흐름 등에 대해 더 깊이 학습하고 적용해볼 수 있었을 것이라는 아쉬움이 남습니다. 이러한 부분까지 충분히 이해하고 설계했다면 코드 구조나 예외 처리 측면에서 더 안정적이고 완성도 높은 결과물을 만들 수 있었을 것이라 생각합니다.

또한 이번 프로젝트에서는 크롤링, 리팩토링, 디자인 개선, 로직 구현 검토 과정에서 AI 도구를 적극적으로 활용했습니다. AI를 통해 코드 구조를 점검하고 개선 방향을 빠르게 확인할 수 있었으며, 익숙하지 않은 Node.js 환경에서도 구현 속도를 높이는 데 큰 도움을 받을 수 있었습니다.
